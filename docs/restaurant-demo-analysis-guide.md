# 레스토랑 데모 분석 가이드 (비개발자용)

레스토랑 데모를 할 때 **무엇을 보고, 어떤 순서로 분석하면 좋은지** 정리한 문서입니다. 코드를 잘 모르셔도 흐름과 포인트만 따라가면 됩니다.

---

## 1. 이 데모가 보여주는 것 (한 줄 요약)

**"사용자가 채팅처럼 질문하면, AI 에이전트가 레스토랑 목록·예약 폼 같은 UI를 JSON으로 보내고, 웹 앱이 그걸 그대로 화면에 그려준다"**가 핵심입니다.

- **텍스트만 주고받는 챗봇**이 아니라
- **에이전트가 만든 폼·카드·버튼**이 화면에 나타나고
- 사용자는 그 UI를 클릭·입력하면서 예약까지 할 수 있습니다.

분석할 때는 **“에이전트 → JSON → 화면”** 이 세 단계가 어떻게 이어지는지 보면 됩니다.

---

## 2. 데모 실행 방법 (분석 전에 한 번 돌려보기)

데모를 직접 켜서 보는 것이 가장 좋습니다.

### Angular 기준 (README 기준)

1. **API 키 준비**  
   - `samples/agent/adk/restaurant_finder/.env.example`을 `.env`로 복사  
   - `.env`에 OpenAI API 키 입력 (`OPENAI_API_KEY=...`)  

2. **한 번에 실행**  
   - 프로젝트 루트에서:  
     `npm install`  
     `npm run demo:restaurant`  
   - 브라우저에서 `http://localhost:4200/` 열기  

### Lit/React 셸을 쓰는 경우

- **Lit**: `samples/client/lit` → `npm run demo:restaurant` 후 `http://localhost:5173/`
- **React**: React 셸이 레스토랑 설정으로 되어 있으면, 에이전트 서버(`restaurant_finder`)를 띄운 뒤 해당 셸 실행

실행되면 **입력창 + 전송 버튼**이 있고, 메시지를 보내면 그 아래에 에이전트가 만든 UI(카드, 버튼, 폼 등)가 나타납니다.

---

## 3. 분석할 때 “무엇을 봐야 할지” 체크리스트

코드보다 **화면과 사용자 흐름**을 중심으로 보시면 됩니다.

### 3.1 사용자 관점 (화면·흐름)

| 볼 것 | 설명 | 분석 포인트 |
|-------|------|-------------|
| **첫 화면** | 검색창 + placeholder 텍스트 | 어떤 문구로 유도하는지(예: "Top 5 Chinese restaurants in New York.") |
| **로딩 중** | 스피너 + 짧은 문구 | "Finding the best spots...", "Checking reviews..." 등 사용자 대기 경험 |
| **레스토랑 목록** | 카드 형태로 레스토랑 이름, 별점, 설명, 이미지, "예약" 버튼 | 카드 구성, 정보 밀도, 버튼 위치가 직관적인지 |
| **예약하기 클릭 후** | 예약 폼(이름, 인원, 날짜/시간, 식이 요구 등) | 폼이 한 화면에 잘 들어오는지, 필수 입력이 뭔지 |
| **예약 제출 후** | 확인 메시지 또는 다음 단계 | 성공/실패 피드백이 명확한지 |

이걸 **스크린샷 + 짧은 메모**로 남기면, “데모에서 무엇을 보여줬는지” 정리하기 좋습니다.

### 3.2 A2UI 관점 (에이전트가 무엇을 “만들었는지”)

데모가 성공적으로 돌아가면, 사실상 아래 세 가지가 동작한 겁니다.

| 볼 것 | 의미 | 분석 시 질문 |
|-------|------|----------------|
| **Surface** | 에이전트가 “한 덩어리 UI”를 붙일 수 있는 영역 | “목록”과 “예약 폼”이 각각 다른 Surface인지, 한 Surface를 갱신하는지 |
| **컴포넌트** | Text, Button, Card, TextField, List 등 | 화면에 보이는 요소가 어떤 “타입”으로 정의돼 있는지 (문서/스펙 참고) |
| **데이터 모델** | 화면에 채워 넣는 실제 데이터(레스토랑 이름, 주소, 사용자 입력값) | “구조(컴포넌트)”와 “데이터”가 분리돼 있는지 |

코드를 안 봐도, **스펙/문서**에서 “createSurface, updateComponents, updateDataModel” 같은 단어가 나오면  
→ “에이전트가 Surface를 만들고, 그 안에 컴포넌트를 넣고, 데이터만 갱신해서 화면을 채운다”라고 이해하시면 됩니다.

### 3.3 메시지 흐름 (분석 보고서용으로만 개념적으로)

| 단계 | 누가 무엇을 하는지 | 분석 시 적어두면 좋은 것 |
|------|-------------------|--------------------------|
| 1 | 사용자가 입력창에 문구 입력 후 전송 | 예: "뉴욕 중국 음식 5곳 찾아줘" |
| 2 | 클라이언트가 에이전트(A2A 서버)로 메시지 전송 | “클라이언트 → 에이전트” 구간이 있다는 것만 인지 |
| 3 | 에이전트가 LLM·도구(get_restaurants 등) 호출 후 **A2UI JSON** 생성 | “에이전트가 UI를 JSON으로 만든다” |
| 4 | 클라이언트가 JSON 수신 후 **화면에 렌더링** | “같은 JSON이라도 Angular/Lit/React 등 다른 클라이언트에서 그릴 수 있다” |
| 5 | 사용자가 버튼(예: 예약) 클릭 → **action** 전송 | “버튼 클릭 = 특정 action 이름 + context(레스토랑 이름 등) 전송” |
| 6 | 에이전트가 action에 맞춰 다음 UI(예: 예약 폼·확인 화면) 생성 후 다시 JSON 전송 | “상호작용 한 번 = 메시지 한 번 오고 가는 구조” |

보고서에 쓸 때는 **“사용자 입력 → 에이전트 처리 → JSON으로 UI 전달 → 화면 갱신 → 사용자 액션 → 다시 에이전트가 UI 갱신”** 정도만 흐름으로 적어도 충분합니다.

- **필요하면 위 문서에서 용어만 찾아보는** 순서를 추천합니다.

## 5. 분석 시 유의할 점 (보안·한계)

- 데모는 **시연용**이며, 외부 에이전트를 “신뢰된 것”으로 가정하지 말라는 문구가 README에 있습니다.
- 실제 서비스라면 **에이전트가 준 JSON·데이터를 검증·제한**해야 한다는 점만 보고서에 짧게 적어두어도 좋습니다.
- “코드에 대해 잘 모른다”면 **구현 디테일보다**  
  - 어떤 화면이 나오는지,  
  - 사용자가 어떤 순서로 조작하는지,  
  - 에이전트와 클라이언트가 “JSON으로 UI를 주고받는다”는 점  
  이 세 가지를 중심으로 정리하시면 됩니다.

---

## 6. 한 페이지 요약 (프린트용)

- **데모 목적**: 에이전트가 JSON으로 UI를 보내면, 클라이언트가 그대로 그려서 레스토랑 검색·예약 흐름을 보여줌.
- **실행**: API 키 설정 후 `npm run demo:restaurant` (또는 사용 중인 클라이언트의 해당 스크립트).
- **볼 것**:  
  1) 첫 화면·로딩·목록·예약 폼·확인 화면(UX),  
  2) Surface/컴포넌트/데이터 모델이 분리된 개념,  
  3) 사용자 입력 → 에이전트 → JSON → 렌더링 → 액션 → 다시 UI 갱신 흐름.
- **참고 문서**: `docs/quickstart.md`, `docs/introduction/what-is-a2ui.md`, `specification/.../a2ui_protocol.md`, 각 샘플의 README.

이 가이드를 기준으로 보시면, 코드 없이도 “레스토랑 데모에서 무엇을 봐야 하는지”를 일관되게 분석하실 수 있습니다.
